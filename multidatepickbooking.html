multiDatePickBooking

Book Resources Across Multiple Dates in Salesforce. Visual time-slot grid with real-time conflict detection for rooms, equipment, vehicles, and people.

Overview

multiDatePickBooking is the most advanced component in the suite. It combines the calendar with a resource selection panel and a visual time-slot grid. Users pick a resource (or multiple resources), select dates, and then click available time slots to create bookings. The component queries your resource object for available items, checks for existing bookings to detect conflicts in real time, and creates booking records automatically.

Use this for conference room reservations, equipment checkout, vehicle scheduling, technician dispatch, or any scenario where a shared resource needs to be reserved on specific dates and times.

Getting Started: Step-by-Step Setup

The Booking component requires two custom objects: a Resource object (what you are booking) and a Booking object (where reservation records are stored).

1 Create the Resource Object

This object stores the things people can book — conference rooms, projectors, vehicles, technicians, etc. Each record is one bookable resource.

Go to Setup > Object Manager > Create > Custom Object. For this example, we'll create Room__c for conference rooms. Name yours whatever fits (Equipment__c, Vehicle__c, Staff__c, etc.).

🖼️Screenshot: Resource object in Setup

2 Create Fields on the Resource Object

The resource object uses the standard Name field plus optional Time fields for business hours:

  • Name (standard field) — display name, e.g., "Conference Room A"
  • Business Hours Start (e.g., Business_Hours_Start__c, type: Time) — when the resource is available from, e.g., 08:00
  • Business Hours End (e.g., Business_Hours_End__c, type: Time) — when the resource is available until, e.g., 18:00

After creating the fields, create a few resource records so the component has resources to show in the dropdown.

🖼️Screenshot: Resource records with business hours

3 Create the Booking Object

This object stores the actual reservations. Each time a user books a time slot for a resource, one Booking record is created.

Go to Setup > Object Manager > Create > Custom Object. Create Room_Booking__c (or Equipment_Reservation__c, Vehicle_Checkout__c, etc.).

4 Create Fields on the Booking Object

  • Booking Date (Booking_Date__c, type: Date) — the date of the reservation
  • Resource Lookup (Meeting_Room__c, type: Lookup to your Resource object) — which resource is booked
  • Start Time (Start_Time__c, type: Time) — when the booking starts
  • End Time (End_Time__c, type: Time) — when the booking ends
  • Parent Lookup (e.g., Travel__c, type: Lookup) — optional, links to the parent record
🖼️Screenshot: Booking object fields

5 Add the Component to a Record Page or Flow

Drag multiDatePickBooking onto your Record Page or Flow Screen and configure the properties below. Make sure to map all the Resource and Booking fields correctly.

🖼️Screenshot: Booking component on a Record Page showing dropdown, calendar, and time grid

Using in a Screen Flow

The Booking component shares Display properties (Button Label, Calendar Title, Show Inline, Single Month View, Calendar Size, Week Starts Monday, Recurring Pattern) and Constraint properties (Max Dates, Allow Past Dates, Min/Max Date) with the other components. See the multiDatePickDates page for those.

The following properties are unique to Booking:

Using on a Lightning Record Page

The Booking component works on Record Pages with the same Resource and Booking configuration properties listed above. The recordId is automatically passed by Salesforce, so bookings are linked to the current record through the Parent Record Relationship Field.

Configure the Resource Object, Booking Object, and all field mappings in the App Builder property panel.

How Conflict Detection Works

When a user selects a resource and dates, the component automatically queries existing booking records for that resource on those dates. For each date, it generates time slots based on the resource's business hours and the configured interval.

If an existing booking overlaps with a generated time slot (the existing booking's start time is before the slot's end time AND the existing booking's end time is after the slot's start time), that slot is marked as unavailable and shown in pink/red on the grid.

When the user clicks the Book button, the system performs one final conflict check before inserting records. Any dates that fail the check are added to the Conflict Dates output and skipped. Successfully booked dates are counted in the Successful Bookings Count output.

For multiple resources, the system queries bookings for ALL selected resources and shows only slots that are free across every selected resource. This ensures that booking a room and a projector together only shows times when both are available.

Using Booking Outputs in Your Flow

The Booking component handles record creation internally — it inserts booking records directly when the user clicks Book. The MultiDatePickParser is not used here. Instead, after the Booking component runs, two output variables tell your Flow what happened: how many slots were successfully booked and which dates had conflicts. Wire them to Flow variables in the component's output properties, then use them in the elements that follow.

Output Variables

Output VariableTypeDescription
bookingSuccessCountIntegerNumber of booking records successfully created. Wire to an Integer variable in your Flow.
bookingConflictDatesString[]Dates where bookings could not be created due to conflicts. Wire to a Text Collection variable. Empty if all bookings succeeded.
💡
Tip: Wire both outputs even if you only plan to use one. Having bookingConflictDates available lets you add conflict handling later without editing the screen element.

Example Use Cases

Here are real-world examples of how to use the Booking component's outputs to drive downstream automation in your Flow:

1 Field Service — Auto-Create Work Orders After Booking

A field service dispatcher books a technician across multiple dates using multiDatePickBooking. After the component runs, bookingSuccessCount confirms how many slots were reserved. The Flow uses bookingSuccessCount in a Decision element — if greater than 0, it enters a Loop that runs bookingSuccessCount times and creates one WorkOrder record per iteration, each linked to the Asset__c and the technician's resource record. The Asset status is updated to "Service Scheduled" after the loop. This pattern turns a visual booking action into a complete work order pipeline without any custom code.

Key outputs used: bookingSuccessCount

Pattern: Booking Component → Decision (count > 0) → Loop → Create Work Orders → Update Asset Status

2 Facilities — Confirmation Email After Room Booking

An employee books a conference room for a series of meeting dates using multiDatePickBooking. After booking completes, the Flow uses bookingSuccessCount in a text formula variable to build a personalized confirmation message: "Your room has been reserved for {bookingSuccessCount} session(s). Please check your calendar for details." The Flow passes this message to a Send Email action, notifying the employee and any attendees. If bookingConflictDates is not empty, a separate email branch notifies the employee which dates could not be booked so they can follow up.

Key outputs used: bookingSuccessCount, bookingConflictDates

Pattern: Booking Component → Build Confirmation Text → Send Email → Conflict Branch (if bookingConflictDates not empty) → Send Conflict Notice Email
